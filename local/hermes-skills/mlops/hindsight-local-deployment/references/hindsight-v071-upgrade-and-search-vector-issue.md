## search_vector GENERATED ALWAYS Column Issue

### Symptom

Retain operations fail with:

```
cannot insert a non-DEFAULT value into column "search_vector"
DETAIL: Column "search_vector" is a generated column.
```

### Root Cause

v0.7.1 changed `search_vector` from a regular column to a `GENERATED ALWAYS` column in PostgreSQL. The Hindsight retain code still attempts to explicitly insert a value into this column, which PostgreSQL rejects.

### Impact

- **Non-blocking**: The failed operations represent a small fraction of total retain work. Most documents are committed before the operation hits this error.
- Evidence from 2026-06-02 daily run: 253 manifest entries (200 production, 51 manual_review, 2 skip) → documents 5654→5708 (+54), observations 53937→54096 (+159). Only 2 retain operations failed with this error.
- Failed operations accumulate in `operations_by_status.failed` (62 as of 2026-06-02, most from this cause).

### Workaround

No code fix needed at the operator level. The data lands successfully despite the operation-level failure. The `search_vector` is auto-generated by PostgreSQL, so the column is still populated correctly for rows that do get inserted.

### Monitoring

When checking daily pipeline success, distinguish:
- **Data-level success**: documents/observations count increased → retain worked.
- **Operation-level failures**: `search_vector` errors → known v0.7.1 issue, non-blocking.
- **True failures**: other error messages (e.g., `generator didn't stop after athrow()`) → investigate individually.

```bash
# Check failed operations and their error messages
curl -s 'http://127.0.0.1:8888/v1/default/banks/hermes/operations?status=failed&limit=10' | \
  python3 -c "
import sys, json
data = json.load(sys.stdin)
ops = data.get('operations', [])
errors = {}
for op in ops:
    msg = op.get('error_message', 'unknown')[:80]
    errors[msg] = errors.get(msg, 0) + 1
for msg, count in sorted(errors.items(), key=lambda x: -x[1]):
    print(f'  {count}x: {msg}')
"
```
