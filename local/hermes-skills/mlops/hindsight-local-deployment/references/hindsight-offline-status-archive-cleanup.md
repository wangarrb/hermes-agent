# Hindsight offline status reporting and archive cleanup notes

Use this reference when the user asks whether observations/consolidation ran, what the offline pipeline does, or whether old Hindsight archives can be cleaned.

## Observation terminology

Always distinguish two different layers:

1. Hindsight native observations/consolidation
   - Controlled by runtime/provider env such as `HINDSIGHT_API_ENABLE_OBSERVATIONS` and `HINDSIGHT_API_WORKER_CONSOLIDATION_MAX_SLOTS`.
   - If these are `false` / `0`, native background observation generation is not running, even if the DB contains `fact_type='observation'` units.

2. Hermes offline V2 canonical observations
   - Generated by local offline scripts from retained facts / daily / weekly summaries.
   - Published back into Hindsight as canonical `memory_units` with `fact_type='observation'` and document ids such as `hermes-offline-canonical::*`.
   - These count as observations in Hindsight stats, but they are not produced by Hindsight's native background consolidation.

When reporting status, say explicitly which layer ran. Example: "Native Hindsight observations are disabled; offline V2 canonical observations have published N units."

## Status checks before answering

Use live checks, not memory:

```bash
python3 ~/.hermes/scripts/hindsight_minimax_import.py status || true
```

DB checks, adapting column names to the current schema:

```sql
SELECT fact_type, count(*)
FROM memory_units
WHERE bank_id='hermes'
GROUP BY fact_type;

SELECT count(*)
FROM memory_units
WHERE bank_id='hermes'
  AND document_id LIKE 'hermes-offline-canonical::%';

SELECT count(*)
FROM memory_units
WHERE bank_id='hermes'
  AND document_id LIKE 'hermes-offline-consolidation::%';
```

Do not assume `documents.document_id` exists; in current Hindsight schema document ids are on `memory_units.document_id`, while `documents` may only have `id`, `metadata`, and text/hash fields.

## Archive cleanup policy

Archive deletion is a destructive operation. First list candidate paths, size, and mtime; do not delete until the user explicitly confirms exact scope.

Typical candidates:
- Current reset rollback points: keep until the current full rebuild, weekly publish, and wiki candidate generation are verified.
  - `~/.hindsight-docker/instances/hindsight/data.pre-reset-*`
  - `~/.hindsight-docker/backups/reset-pre-full-rebuild-*`
- Older backups/archives: can be proposed for deletion after verification.
  - `~/.hindsight-docker/backups/backup-*`
  - `~/.hermes/hindsight/offline_reflect_archive`

Recommended listing command:

```bash
python3 - <<'PY'
from pathlib import Path
import subprocess, time
roots = [Path('~/.hindsight-docker/backups').expanduser(), Path('~/.hindsight-docker/instances/hindsight').expanduser(), Path('~/.hermes/hindsight').expanduser()]
keys = ['backup','bak','pre-reset','archive','old','disabled','migration','migrat']
seen = []
for base in roots:
    if not base.exists():
        continue
    for p in base.iterdir():
        if p.is_dir() and any(k in p.name.lower() for k in keys):
            seen.append(p)
for p in seen:
    size = subprocess.check_output(['du','-sh',str(p)], text=True).split('\t')[0]
    mt = time.strftime('%Y-%m-%d %H:%M:%S', time.localtime(p.stat().st_mtime))
    print(f'{size}\t{mt}\t{p}')
PY
```

If the user asks to clean "old migrated Hindsight archives", propose keeping the latest reset rollback artifacts and deleting only older clearly superseded archives, unless they explicitly authorize more.

## Cron reporting pitfall

Report the actual cron expression, not only the prompt wording. If prompt says "biweekly" but cron expression is `0 5 * * 0`, the actual cadence is weekly. Call out the mismatch and offer to correct it.
